## Storage Systems(Which databases/storage systems did you choose for each of the 4 goals and why)

The architecture uses a **polyglot persistence** approach, matching each storage engine to its specific workload.

1. PostgreSQL serves as the primary OLTP database for structured patient records, admission history, and treatment data. Its ACID compliance and rich relational querying make it the natural choice for transactional clinical data that requires strict consistency — for example, when a doctor updates a patient's medication record while a readmission risk score is being computed from the same row.

2. MongoDB handles unstructured and semi-structured data such as clinical notes, discharge summaries, and HL7 FHIR JSON documents. Its flexible schema accommodates the variability inherent in medical documentation without requiring costly migrations when note formats change.

3. InfluxDB is dedicated to the real-time ICU vitals stream. It is purpose-built for high-frequency time-series data (e.g., heart rate sampled every second across hundreds of beds) and supports efficient downsampling and retention policies that would be prohibitively expensive in a relational engine.

4. Snowflake / BigQuery forms the OLAP data warehouse. Structured data from PostgreSQL is replicated via Change Data Capture (CDC) into the warehouse, where dbt models produce pre-aggregated marts for reporting (bed occupancy, department costs) and a Feature Store for ML model training.

5. AWS S3 provides object storage for raw medical imaging files, exported reports, and serialised ML model artefacts. A *pgvector* extension on PostgreSQL stores dense embeddings generated by the NLP query engine, enabling fast semantic similarity search for the doctor-facing RAG pipeline.


## OLTP vs OLAP Boundary(Where does the transactional system end and the analytical system begin in your design)

1. The OLTP boundary ends at PostgreSQL and MongoDB. These systems handle every write that originates from clinical workflows — admissions, lab results, nursing charting — where low-latency writes and row-level consistency are required. InfluxDB also sits in the OLTP tier because ICU devices write continuously at high frequency.

2. The OLAP boundary begins with Snowflake. A CDC pipeline (e.g., Debezium) captures row-level change events from PostgreSQL and replicates them asynchronously into the warehouse, where dbt transformations denormalise and aggregate the data into reporting-friendly models. This separation is critical: running heavy analytical queries (e.g., monthly bed-occupancy aggregations spanning millions of rows) directly on the transactional database would degrade response times for live clinical users.

The Feature Store sits at the OLAP layer — it materialises pre-computed features (e.g., 30-day readmission counts, average length-of-stay by diagnosis) that the ML models consume at training time, without touching the live OLTP tables.


## Trade-offs(Identify one significant trade-off in your design and how you would mitigate it)

1. The most significant trade-off in this design is consistency versus analytical freshness. Because the OLAP warehouse is populated asynchronously via CDC, there is an inherent replication lag (typically seconds to minutes). Monthly management reports and ML training jobs therefore operate on data that is slightly stale relative to the live OLTP system. This is acceptable for batch analytics but would be unsuitable for real-time clinical decision support.

2. To mitigate this, the ICU anomaly-detection model bypasses the warehouse entirely and consumes directly from the Kafka stream, trading the richness of historical feature data for sub-second latency. This introduces a second trade-off: the streaming model is trained on a narrower feature set than the batch readmission model, potentially reducing its predictive accuracy. The design accepts this limitation because the primary requirement for ICU alerts is timeliness, not exhaustive feature engineering.

3. A secondary trade-off is operational complexity. Running six distinct storage technologies (PostgreSQL, MongoDB, InfluxDB, Snowflake, S3, pgvector) increases infrastructure cost and requires a broader range of engineering expertise. A simpler design using only PostgreSQL and Snowflake would reduce overhead but would not meet the latency requirements of the real-time vitals stream or the semantic search requirements of the NLP query engine.
